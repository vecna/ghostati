Produce or extend a JSON reference set for the same project universe as the existing file "REFERENCES.json".

Goal:
Maintain a coherent, deduplicated set of cultural, artistic, academic, and activism-adjacent references around:
- anti-surveillance aesthetics
- face obfuscation / face camouflage
- makeup-based or appearance-based interventions
- adversarial attacks on face recognition
- physical-world adversarial vision techniques adjacent to facial recognition
- sensor-specific attacks that help explain why different capture technologies require different countermeasures

Output rules:
1. Output valid JSON only.
2. Preserve the top-level structure:
{
  "schema_version": "2.1",
  "project_scope": "...",
  "ordering_definition": "...",
  "language": "en",
  "closeness_definition": {...},
  "tag_definitions": {...},
  "references": [...]
}
3. Each reference object MUST contain at least:
- slug
- title
- author
- year
- link
- links
- type
- closeness
- description
- demonstrated
- ghostati_relevance
- ghostati_testability
- intervention
- target
- domain
- limitations
- reproducibility
- citation
4. All editorial text MUST be written in English.
5. Preserve existing slugs for existing references.
6. Return the full updated JSON, not just the new or changed entries.

Field conventions:
- slug: kebab-case unique identifier stable across updates.
- title: canonical public title of the work/project/paper.
- author: array of author/creator names in display order.
- year: integer year of publication, release, first public presentation, or first documented appearance.
- period: optional human-readable range, for projects with a multi-year lifespan such as "2019–2021".
- link: stable canonical link preferred in this order: DOI > arXiv abs page > official project page > official institutional page.
- links: object with canonical, paper, project, code, video, doi, arxiv. Use null for unknown or unavailable fields.
- type: exactly one of "artistic" | "research" | "activism".
- closeness: integer from 1 to 100, not a string and no percent sign.
- description: short editorial description for the public references page, 1–2 sentences.
- demonstrated: concise technical or curatorial summary of what the work actually showed, built, tested, or performed.
- ghostati_relevance: why this reference matters to Ghostmaxxing specifically.
- ghostati_testability: object with:
  - level: exactly one of "direct" | "partial" | "indirect" | "contextual" | "no"
  - reason: short explanation of what Ghostmaxxing can or cannot retest.
- intervention: array of normalized tags describing the technique or practice.
- target: array of normalized tags describing the system or pipeline stage affected.
- domain: array of normalized tags describing the operational domain.
- limitations: explicit limitations. Use stated limitations when available; infer cautiously when needed. Do not overclaim.
- reproducibility: short normalized phrase such as "paper-available", "code-available", "documented-project", "artwork-documentation", "survey", "unclear".
- citation: short display citation such as "Yin et al., IJCAI 2021" or "Adam Harvey, CV Dazzle, 2010".
- visual_hint: optional short internal hint used to generate or select a preview image.

Closeness rubric:
- 100 = exact anchor cluster for this project world:
  * Michelle Tylicki + Lauri Love — DAZZLE
  * Adam Harvey — CV Dazzle
- 90–99 = direct face-centered anti-surveillance / makeup / camouflage references.
- 70–89 = directly relevant face-recognition attack literature or highly adjacent face-obfuscation work.
- 40–69 = enabling or adjacent adversarial-vision methods that matter conceptually but are less face/makeup-specific.
- 1–39 = peripheral or modality-shifted references, especially infrared/night-vision/sensor-jamming examples.

Ghostmaxxing testability rubric:
- direct: Ghostmaxxing can retest a closely similar browser-visible face intervention or matching/detection behavior.
- partial: Ghostmaxxing can approximate one part of the intervention, but not the full experimental setup.
- indirect: Ghostmaxxing can use the reference conceptually, but needs different sensors, models, or optimization to validate it.
- contextual: the reference helps explain history, public practice, aesthetics, or methodology, but is not directly testable.
- no: the reference is relevant to the wider map but cannot be tested with Ghostmaxxing’s browser AR pipeline.

Tag rules:
- Tags MUST be lowercase kebab-case.
- Prefer existing tags from tag_definitions.
- If adding a new tag, also add it to tag_definitions.
- Do not create near-duplicates such as "face-recognition" and "facial-recognition".

Research and limitation rules:
- Read the canonical source before adding or changing demonstrated, limitations, links, or testability.
- Use primary sources where possible: DOI page, arXiv abstract page, official project page, official repository, institutional page.
- Do not invent missing metadata.
- If limitations are not explicit, infer conservatively and phrase them as scope limits.
- Good limitation pattern: "The reported result depends on the tested model family, acquisition setup, physical realization, and subjects. Effectiveness against newer production systems is not established by this reference."
- Avoid: "This works", "This defeats facial recognition", "This proves protection".

Normalization rules:
- Deduplicate by normalized title first, then by DOI or canonical link.
- Prefer canonical capitalization.
- Prefer full author lists when available.
- Keep the existing entry if the new candidate is the same work with a worse link.
- If the same work exists with a better canonical link, replace only the link and preserve closeness unless there is a strong reason to revise it.
- Preserve the existing slug whenever the same work reappears.

Ordering logic:
- Return the full updated JSON sorted newest first:
  year descending, closeness descending, title ascending.
- references.html is generated newest-first, grouped into decade sections; there is no interactive toggle to reverse the order (the old filter/genealogy-mode UI was removed — see scripts-dev/build-references-page.js).

Style constraints:
- Avoid generic press coverage unless it is itself the primary artistic or activist source.
- Prefer official project pages, DOI links, arXiv abs pages, institutional pages, or artist pages.
- Keep the set operational: every added item should help situate the project culturally, artistically, academically, or politically.
- Descriptions should be concise, readable, and suitable for a static references page.
- Every entry should help answer:
  1. What is it?
  2. What did it demonstrate?
  3. What technique did it use?
  4. What are its limits?
  5. Can Ghostmaxxing directly or partially retest it?
  6. Where can people read more?
